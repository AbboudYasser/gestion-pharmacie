# دليل اختبار نظام إدارة الصيدلية 🧪

يهدف هذا الدليل إلى توضيح كيفية اختبار نظام إدارة الصيدلية محلياً وعلى بيئة النشر (GitHub Pages) لضمان عمل جميع المكونات بشكل صحيح.

## 1. الاختبار المحلي (Local Testing)

لضمان عمل الواجهتين الأمامية والخلفية معاً قبل النشر، يمكن تشغيلهما محلياً.

### 1.1 تشغيل الواجهة الخلفية (Backend)

تعتمد الواجهة الخلفية على Python Flask وتتطلب بعض المتطلبات.

#### المتطلبات:
- Python 3.8+
- `pip` (مدير حزم Python)
- ملف `.env` في مجلد `backend` يحتوي على متغيرات البيئة اللازمة (مثل `FIREBASE_PROJECT_ID`, `FIREBASE_PRIVATE_KEY`, `SECRET_KEY`, `JWT_SECRET_KEY`).

#### خطوات التشغيل:

1.  **الانتقال إلى مجلد الواجهة الخلفية:**
    ```bash
    cd backend
    ```

2.  **تثبيت المتطلبات:**
    تأكد من تثبيت جميع الحزم المطلوبة. إذا لم تكن مثبتة، يمكنك تثبيتها باستخدام:
    ```bash
    pip install -r requirements.txt
    ```
    _ملاحظة: إذا واجهت مشاكل في `firebase-admin`، تأكد من أن `private_key` في ملف `.env` منسق بشكل صحيح (يجب أن تكون علامات `\n` موجودة كأحرف فعلية لكسر السطر)._

3.  **تشغيل الخادم:**
    ```bash
    python main.py
    ```
    يجب أن ترى رسالة تفيد بأن خادم Flask يعمل على `http://127.0.0.1:5000` أو `http://0.0.0.0:5000`.
    
    _ملاحظة: إذا لم يتم تهيئة Firebase بنجاح، سيتم استخدام قاعدة بيانات مؤقتة في الذاكرة. هذا يسمح للواجهة الخلفية بالعمل ولكن بدون حفظ البيانات بشكل دائم._

4.  **اختبار نقطة نهاية Health Check:**
    افتح terminal جديد واختبر API:
    ```bash
    curl http://localhost:5000/api/health
    ```
    يجب أن تحصل على استجابة JSON مثل:
    ```json
    {
      "message": "نظام إدارة الصيدلية يعمل بشكل صحيح",
      "status": "healthy"
    }
    ```

### 1.2 تشغيل الواجهة الأمامية (Frontend)

الواجهة الأمامية هي تطبيق HTML/CSS/JavaScript بسيط يمكن تشغيله باستخدام خادم HTTP بسيط.

#### المتطلبات:
- Python 3.8+ (لخادم HTTP البسيط)

#### خطوات التشغيل:

1.  **الانتقال إلى المجلد الرئيسي للمشروع:**
    ```bash
    cd /path/to/your/project/root
    ```

2.  **تشغيل خادم HTTP:**
    ```bash
    python -m http.server 3000
    ```
    يجب أن ترى رسالة تفيد بأن الخادم يعمل على `http://0.0.0.0:3000`.

3.  **فتح الواجهة الأمامية في المتصفح:**
    افتح متصفح الويب الخاص بك وانتقل إلى `http://localhost:3000`.
    يجب أن تظهر لك الصفحة الرئيسية لنظام إدارة الصيدلية.

### 1.3 اختبار وظائف الواجهة الأمامية (Manual Testing)

بعد تشغيل الواجهتين، يمكنك اختبار التفاعل بينهما:

1.  **الانتقال إلى صفحة تسجيل الدخول:**
    من الصفحة الرئيسية، انقر على زر "ابدأ الآن" أو انتقل مباشرة إلى `http://localhost:3000/frontend/login.html`.

2.  **محاولة تسجيل الدخول:**
    - أدخل اسم مستخدم (مثال: `testuser`)
    - أدخل كلمة مرور (مثال: `password123`)
    - اختر دوراً (مثال: `مدير المستشفى`)
    - انقر على زر "تسجيل الدخول".

    _ملاحظة: بما أن Firebase قد لا يكون مهيئاً بشكل كامل محلياً، فإن الواجهة الأمامية ستستخدم البيانات الوهمية (Mock Data) تلقائياً إذا فشل الاتصال بالخادم الخلفي. هذا يسمح لك باختبار تدفق تسجيل الدخول والانتقال إلى لوحة التحكم حتى بدون خادم خلفي يعمل بكامل طاقته._

3.  **التحقق من لوحة التحكم:**
    بعد تسجيل الدخول، يجب أن يتم توجيهك إلى `dashboard.html`.
    تحقق من عرض البيانات (حتى لو كانت وهمية) وتفاعل مع عناصر الواجهة.

## 2. اختبار النشر على GitHub Pages

يتم اختبار النشر تلقائياً بواسطة GitHub Actions بعد كل عملية `push` إلى فرع `main`.

### 2.1 مراقبة GitHub Actions

1.  **الانتقال إلى صفحة Actions:**
    في مستودع GitHub الخاص بك، انتقل إلى علامة التبويب `Actions`.

2.  **مراقبة Workflow `Deploy to GitHub Pages`:**
    ستجد workflow باسم `Deploy to GitHub Pages` قيد التشغيل أو تم تشغيله مؤخراً.
    انقر على أحدث تشغيل لمشاهدة التفاصيل.

3.  **مراجعة خطوات النشر:**
    - **Build Project**: يتم بناء المشروع وتحضير الملفات للنشر.
    - **Configure for Production**: يتم تحديث `config.js` لعنوان API الخاص بالإنتاج وتفعيل البيانات الوهمية.
    - **Validate Build**: يتم التحقق من وجود جميع الملفات الأساسية.
    - **Deploy to GitHub Pages**: يتم نشر الملفات إلى GitHub Pages.
    - **Post-Deploy Tests**: يتم إجراء اختبارات بسيطة (مثل `curl`) للتأكد من أن الصفحات الرئيسية قابلة للوصول بعد النشر.

### 2.2 التحقق من الموقع المنشور

بعد اكتمال workflow بنجاح، يمكنك زيارة الموقع المنشور:

1.  **الحصول على رابط الموقع:**
    من صفحة `Actions`، في مهمة `Deploy to GitHub Pages`، ستجد رابط `page_url`.
    عادة ما يكون الرابط على الشكل التالي:
    `https://<your-username>.github.io/<repository-name>/`

2.  **اختبار الصفحات:**
    - افتح الرابط في متصفحك.
    - حاول الوصول إلى صفحة تسجيل الدخول: `https://<your-username>.github.io/<repository-name>/frontend/login.html`.
    - حاول تسجيل الدخول (باستخدام البيانات الوهمية، حيث أن الواجهة الخلفية لن تكون متاحة على GitHub Pages).

### 2.3 فحص جودة الكود (Code Quality Check)

يتم تشغيل workflow آخر باسم `Code Quality Check` تلقائياً على كل `push` و `pull request`.

1.  **مراقبة Workflow `Code Quality Check`:**
    في علامة التبويب `Actions`، ابحث عن هذا workflow.

2.  **مراجعة تقرير الجودة:**
    - **File Structure Check**: يتأكد من وجود جميع الملفات الأساسية.
    - **HTML Validation**: يفحص صحة HTML.
    - **CSS Validation**: يفحص صيغة CSS.
    - **JavaScript Validation**: يفحص صيغة JavaScript.
    - **Python Validation**: يفحص صيغة Python ونمطه.
    - **Security Check**: يبحث عن ملفات حساسة أو أسرار مبرمجة.
    - **Link Check**: يتأكد من صحة الروابط الداخلية.
    - **Quality Report**: يقدم ملخصاً لنتائج جميع الفحوصات.

    هذا Workflow يساعد في الحفاظ على جودة الكود ويمنع دمج التغييرات التي تحتوي على أخطاء أو مشاكل هيكلية.

## 3. توثيق عملية النشر

تم توثيق عملية النشر وإدارة الفروع بشكل مفصل في ملف `docs/deployment-guide.md`.

### النقاط الرئيسية في دليل النشر:

-   **استراتيجية الفروع**: شرح مفصل لاستخدام فروع `main`, `develop`, `feature/*`, `hotfix/*`.
-   **عملية النشر**: خطوات النشر التلقائي عبر GitHub Actions.
-   **إعداد GitHub Repository**: كيفية تفعيل GitHub Pages، إعداد حماية الفروع، وإدارة GitHub Secrets.
-   **التكوين المحلي**: إرشادات لإعداد بيئة التطوير المحلية.
-   **مراقبة النشر**: كيفية تتبع حالة النشر واستكشاف الأخطاء.
-   **أفضل الممارسات**: إرشادات لرسائل الـ commit، قوالب Pull Request، ومراجعة الكود.

يرجى مراجعة هذا الملف للحصول على فهم شامل لعملية النشر وإدارة المشروع.
